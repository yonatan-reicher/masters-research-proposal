\section{Problem Statement}
In the intersection between the worlds of mathematical theory, formal logic,
and programming, exists a species of tools called proof assistants.
These are programming languages, meant not for building software, but for
building mathematical proofs.
Proof assistants, contrary to pen and paper, check formalized proofs for
correctness and soundness.

These so called proof assistants admit only entirely formalized proofs, where
every step is fully justified based on a complex logical semantics sitting on
top of a complex declarative semantics. That is to say, not only are the logical
steps themselves complex and rigorous, but so is the way you express your proof
and how the steps are applied.

For many years, proof assistants were a niche, unpopular family of tools, used
mostly by experts and enthusiasts. However, one member of that family in
particular has been gaining traction in the mathematical community, bringing
many inexperienced users to this world of expert-made, technical and rigorous
proofs.

\lean (\todo{cite?}) is an interactive proof assistant geared towards a general
math audience.
Two things that make it especially appealing are its large ecosystem and its
intuitive syntax.
For example, it has a large library of formalized mathematics called
\mathlib, which contains many foundational and advanced mathematical
results and theories, and includes notation familiar to mathematicians.
Despite that, there are some challenges in adopting \lean. One challenge in
particular, is how complex \lean's semantics and type system are.

\section{Existing Solutions}
When first wanting to formalize a proof in \lean, one usually takes human
mathematicians and sends them to learn the language, and write their proof. This
solution comes with the obvious drawback of being slow, requiring a lot of human
time and effort. \lean has a steep learning curve, and it can take months to
learn the basics.

A more automated approach is to formalize just the theorems and definitions, and
then use an automated theorem prover to fill in the gaps. Several such tools
exist, such as \aesop, \grind, and \leanegg (\todo{cite..?}).
These tools complete a theorem, or part of a theorem, using various heuristics
and algorithms.
They tend to be integrated into \lean, and more fit to use as part of
interactive proof development, rather than full automation.
Moreover, they tend to work only on simple theorems, fail without human guidance,
and even when they succeed, they do not produce readable a proof.

For a more wholistic approach for formalizing a proof, one can reach for any
off-the-shelf generative large language model such as \chatgpt.
In an agentic setting, these systems have gotten good at generating idiomatic
and correct \lean code.
One can write their paper, provide it to the agent, and prompt it to generate a
formal \lean proof.
One can even interact with it to steer towards better directions.
There are even tools in place to help large language models work with \lean such
\aristotle (\todo{cite?}).
These systems suffer a number of drawbacks. To name a few:
They lack reproducibility,
are usually based on cloud services,
and are not fit with an interactive and iterative workflow.

There are also libraries for making proofs easier to read and write. One example
is \waterproof. This is a library for the \rocq proof assistant, which adds
notation inspired by pen-and-paper proofs.
An example proof, written with and without \waterproof respectively, can be seen
in figures \ref{fig:waterproof} and \ref{fig:no-waterproof}.
It does require the proof to be
fully formalized, and informal proofs would require major work to be adapted to
\waterproof.

\section{Our Approach}
We define a language for writing \emph{informal} proofs that can be partially
translated to \lean, to help with starting a formal proof.
The language is a modification of markdown, with some syntax constructs being
recognized as formal. All parts are readable as they are written.
The language can be interfaced via \lean directly by a macro,
and can access \lean and \mathlib symbols.
It is meant as a way to write an informal proof to be understood, and a formal
proof to be checked.

This workflow is in accordance to how researchers tend to use \lean, as a formal
proof in \lean often comes as a supplementary part of a paper proof inside a
research paper.

\begin{figure}
    \include{example}
    \caption{Markdown text, and \lean code that our tool generates}
    \label{fig:example}
\end{figure}

One difference between our tool and other approaches, is that ours allows an
informal proof that is only partially translated. By only aiming to extract as
much information as we can from the user, without aiming for a complete
rigorous proof, the tool can have less friction both in formalization and in
notation.

The tool is invoked by simply calling the macro \code{define\_latex file "foo.md"},
with \code{foo.md} replaced with the file name of the markdown document of your
choice. It is also possible to use the macro with a string literal instead of a
file by omitting the "file" keyword: \code{define\_latex "Hello \$x=world\$"}.

See example in figure \ref{fig:example}.

proof of concept. We are using a static analysis to choose between different
possible representations.

\paragraph{Case Study}
The choose the following problem as our subject for the initial exploration of
the design space:

\begin{center}
\fbox{
    \parbox{0.7\textwidth}{
        Can you cover a $10 \times 10$ board with $1 \times 2$ tiles, such that the
        number of horizontal and vertical tiles is the same?
    }
}
\end{center}

As it turns out, you can't. One can prove that it is impossible. A short
intuitive proof can be seen in figure \ref{fig:conjectures}.
While this proof is readable and correct, and even relatively formal, it does
not translate well to \lean.
As an example, here is a part of the proof that represents conjecture 3.

\begin{verbatim}
  lemma disjoint_hvtiles (tiling : DominoTiling n m) :
    Disjoint (htiles tiling) (vtiles tiling) := by
    unfold htiles vtiles
    apply Finset.disjoint_filter_filter'
    apply Pi.disjoint_iff.mpr
    intros
    unfold horizontal vertical
    split <;> simp
\end{verbatim}

In general, most lemmas do not translate directly. Logical steps that most
mathematicians see as obvious must be stated explicitly, via various different
keywords and auxiliary lemmas.
Many times, it is hard to find a way to translate the logical steps to \lean's
keywords and \mathlib's lemmas, and there isn't a one to one correspondence.

Another challenge, is the representation of constructs and various mathematical
objects. Just in our example, there were various objects that are not obvious to
state in a formal way: What is a board? What is a tile?
For the proof in figure \ref{fig:conjectures}, we choose a specific
representation based on sets.
For \lean, that is not specific enough.
One might interpret that as any of the following.

\begin{leancode}
def Domino (p1 p2 : Point) : Set Point := \{ p1, p2 \}

def Domino (p1 p2 : Point) : Finset Point := \{ p1, p2 \}

inductive Domino \\
\tab | H (lft : Point) \\
\tab | V (top : Point)

structure Domino where \\
\tab topLeft : Point \\
\tab isHorizontal : Bool

def Domino n m := \{ set // isDomino n m set \} \\
where isDomino n m (set : Finset Point) := \\
\tab ($\forall$ p1 p2, p1 $\in$ set $\to$ p2 $\in$ set $\to$ p1.distance p2 $\le$ 1) \\
\tab $\land$ set.card = 2 \\
\tab $\land$ ($\forall$ p $\in$ set, 0 $\le$ p.x < n $\land$ 0 $\le$ p.y < m)
\end{leancode}

Each of the above representations will have consequences on how the proof is
structured, how hard it will be to formalize, and on the representation of all
of the rest of the proof. The \code{Set} type is of sets, \code{Finset} is the
type of finite sets. Even the difference between these two representations
can affect the entire proof.

\paragraph{The Tool}
Our tool takes markdown file as input and outputs definitions directly into the
environment of the lean file that calls it. When the macro is interpreted by the
compiler, the tool reads the given input markdown and runs the following
pipeline:

\begin{enumerate}
    \item Spanning - all math elements (delimited by \$ or by \$\$) from the
        document are extracted.
    \item Lexing - math elements are broken into lists of tokens.
    \item Paring - each token list is parsed into an Ast based on a grammar for
        the tool's language.
    % \item Categorizing - Asts are analyzing
    \item Analyzing - the Ast is translated into a datalog
\end{enumerate}
In the Lexing and Parsing stages, errors are ignored and their math elements are
silently forgotten. This allows the user to write down formulas that will simply
be ignored.

| # | Stage         | Output        | Representation            | Extra Info   |
| - | ------------- | ------------- | ------------------------- | ------------ |
| 1 | Input         | Text          | Array (Array Char)        |              |
| 2 | Spanning      | Inline Math   | Array (Kind ✗ Span)[^1]   |              |
| 3 | Lexing        | Tokens        | Array (Kind ✗ List Token) |              |
| 4 | Parsing       | Ast           | Array (Kind ✗ Formula)    |              |
| 5 | Categorizing  | Category      | Array CategorizedFormula  |              |
| 6 | Analysing     | Analysis      | Analysis ✗ Array ...      |              |
| 7 | Translation   | Lean Commands | Array LeanCmd             |              |
| 8 | Emitting      | Lean Commands | CommandElabM Unit         |              |


As a proof of concept, we made a tool which compiles inline math in a markdown
document to \lean code, and uses a static analysis of the document to determine
which representation to use for each mathematical object.
You can invoke the tool interactively inside a \lean document, giving it a
markdown document, and it declares variables and definitions which can be used
from that point on as if they were declared in the file itself directly.

The tool we have right now is very basic, and the only possible representations
are \code{Set} or \code{Finset} for sets, and \code{Multiset} for multisets and
\code{Prod} for tuples.
